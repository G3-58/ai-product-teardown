## 核心框架：Agent = 规划 + 记忆 + 工具

```
                    ┌─────────────┐
                    │   Agent     │
                    │  Core LLM  │
                    └──────┬──────┘
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
       Planning         Memory          Tool Use
      (规划模块)        (记忆模块)       (工具调用)
```

### 规划模块（Planning）

**任务分解**：
- CoT（Chain-of-Thought）：思维链，逐步推理
- ToT（Tree-of-Thought）：树状探索多条路径，选最优
- Plan-and-Solve：先生成计划再执行，减少「走一步看一步」的错误累积

**自我反思（Self-Reflection）**：
- ReAct：推理和行动交替
- Reflexion：失败后用自然语言反思错误，下次避免
- 关键洞察：LLM 有能力评估自己的输出，这是自我改进的基础

### 记忆模块（Memory）

Weng 用神经科学类比把 LLM 的记忆分四类：

| 类型 | 对应人类记忆 | LLM 实现 |
|------|------------|---------|
| 感觉记忆 | 短暂感知（<1s）| 原始输入的 embedding 表示 |
| 短期记忆 | 工作记忆（~30s）| In-context window |
| 长期记忆 | 语义/事件记忆 | 外部向量数据库（RAG） |
| 程序记忆 | 技能记忆 | Fine-tuning |

**对我的启发**：这个分类框架让我第一次想清楚了为什么 RAG 是「长期记忆」而不是「知识注入」——Agent 本身不「记住」知识，只是在需要时去外部存储检索，跟人类用外部笔记本是一个道理。

### 工具调用（Tool Use）

**关键论文**：
- MRKL System：给 LLM 接不同的专家模块（计算器、搜索引擎、数据库）
- Toolformer：让模型自己学什么时候调用工具，而不是靠规则
- HuggingGPT：把 HuggingFace 上的各种模型当工具，LLM 做任务规划和结果整合

## 案例分析：AutoGPT 的失败模式

文章分析了 AutoGPT 的问题，这对理解 Agent 产品设计很有价值：

**AutoGPT 的逻辑**：给它一个目标，它自己分解、自己执行、自己检查。

**为什么用户最终放弃**：
1. **错误累积**：每步的小误差叠加，最后跑偏很远
2. **不透明**：用户看不懂 Agent 在做什么，没有掌控感
3. **死循环**：遇到模糊情况会无限重试同一个工具
4. **成本无上限**：一个任务跑几十轮 API 调用，费用不可控

**产品教训**：完全自主的 Agent 不是好的用户体验。用户需要的是「自动化的效率 + 人类的控制感」，而不是「把任务完全交出去然后祈祷」。

## 与现在（2025年）的对比

Weng 的文章写于 2023 年，现在变化最大的地方：

| 问题 | 2023年的方案 | 2025年的进展 |
|------|------------|------------|
| Context 太短 | 用摘要压缩 | 128K+ 窗口已普及 |
| Tool use 不稳定 | 大量 prompt engineering | Function Calling 原生支持 |
| 多 Agent 协作复杂 | 手写状态机 | LangGraph / AutoGen 框架化 |
| 评测困难 | 人工评测 | LLM-as-a-Judge 部分自动化 |

Context 变长解决了很多问题，但规划和记忆的核心挑战没有变——LLM 依然不擅长 100 步以上的长程规划。

---

## 一句话总结

> Agent = 规划 + 记忆 + 工具，三者缺一会有明确的失效模式。产品设计的核心是在「自动化效率」和「用户掌控感」之间找到平衡点。
